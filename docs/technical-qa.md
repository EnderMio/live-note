# 技术问答

## 为什么现在只保留 Tk GUI，而不再维护 Qt 预览链路？

项目当前的核心目标是把正式产品链路做稳：真实录音、导入、会话恢复、Obsidian、LLM、离线精修和任务队列都需要围绕同一套界面持续收口。继续长期维护一套 Qt 预览会分散设计、测试和文档成本，也会让仓库同时背负两套桌面叙事。现阶段先收敛到一套可用 GUI，比同时保留一套高保真预览更重要。

## 为什么把导入、重转写、离线精修、合并做成持久任务队列？

这些动作都可能持续数分钟，而且用户常常会连续排多个任务。队列现在统一收口到 `.live-note/runtime-v2/control.db` 的 `tasks` 表，保证 FIFO 顺序、关闭窗口后可恢复、重复任务可去重，也避免同时跑太多离线动作把机器压满。

## 队列在异常退出后如何恢复？

启动 GUI 时会启动 `RuntimeHost` 并恢复 `control.db`。`queued` 任务保留并继续排队；上次处于 `running` 的任务会标记为“中断”并移出待执行任务，不会自动重跑，避免对用户已经半途终止的任务做错误重试。

## 为什么旧 live 会话没有 `session.live.wav` 也能做离线精修？

很多早期会话虽然没有保留整场录音，但仍保存了 `segments.jsonl` 和每段 `segments/*.wav`。现在会先按分段时间轴回放这些 wav，并在片段之间补静音，重建出一条可供 whisper.cpp 再处理的整段音频，再进入离线精修流程。

## 这是不是无损恢复原始录音？

不是。它是“可用于再次转写”的工程恢复，不是司法取证式还原。能恢复的是片段内容和时间轴，不保证保留原始底噪、重叠、设备时钟漂移或未落盘的瞬间音频。

## 为什么历史页底部按钮改成自适应换行？

历史页动作多，而且不同平台字体宽度不一致。固定单行在窄窗口下会溢出，固定两行又会在宽窗口里浪费空间。自适应换行能按可用宽度重新排布，保证按钮文字完整可见。

## 为什么设置与诊断页此前滚轮无效，现在如何工作的？

原先只有滚动条，没有把鼠标滚轮事件路由到内层 `Canvas`。现在用 `Canvas + Scrollbar + bind_all` 组合，并在事件处理里判断指针是否位于该滚动区域上方，只有命中当前页面时才消费滚轮，从而支持触控板、鼠标滚轮和滚动条三种方式。

## 为什么品牌资源要同时保留 SVG 和 PNG？

SVG 用于 README、文档和其他需要无损缩放的场景；PNG 用于 Tk 窗口图标，因为 Tk 对位图更稳定。两者共用同一套图形语义，但分别服务不同渲染需求。

## 为什么 Makefile 要优先使用 `.venv/bin/python`？

项目依赖通常只装在仓库自己的虚拟环境里。如果 `make` 仍调用系统 `python3`，会出现“之前能跑、现在找不到依赖”的假故障。优先使用 `.venv/bin/python` 能减少这类环境偏差。
